# 2048-New-era

一个从近乎空仓启动的高质量 `2048` 项目蓝图。当前仓库几乎没有业务代码，因此本 README 主要承担三件事：

1. 明确项目定位与完成标准
2. 给出适合多 agent 并行协作的实施计划
3. 作为后续开发、检查、修复的统一上下文

## 当前状态

- ✅ 已完成 `Vite + React + TypeScript` 脚手架
- ✅ 已完成核心游戏引擎（纯函数，52 个引擎单元测试）
- ✅ 已完成 UI 层（棋盘、分数、弹层、响应式样式）
- ✅ 已完成交互层（键盘方向键/WASD、移动端触摸滑动、120ms 移动锁）
- ✅ 已完成动画（tile 出现、合并、分数 bump）
- ✅ 已完成 `localStorage` 最高分持久化
- ✅ 已有 76 个自动化测试覆盖引擎、组件、hook、存储
- ⏳ 待完成：E2E 测试、无障碍优化、PWA、README 部署说明

## 项目目标

打造一个体验完整、视觉精致、逻辑可靠、便于扩展的新版 `2048` 游戏，至少满足以下要求：

- 支持桌面端与移动端
- 核心移动与合并规则完全正确
- 有分数、最高分、重新开始、失败检测、胜利提示
- 有平滑动画、清晰反馈和统一视觉风格
- 有可维护的代码结构和自动化测试

## 推荐技术路线

为减少分歧，建议统一采用以下技术栈：

- 前端：`Vite + React + TypeScript`
- 样式：原生 `CSS` 或 `CSS Modules`，使用设计变量管理颜色、圆角、阴影和动画时长
- 状态管理：`useReducer` 或独立的纯函数游戏引擎
- 单元测试：`Vitest`
- 组件测试：`React Testing Library`
- 端到端测试：`Playwright`
- 代码质量：`ESLint + Prettier`

如果后续要改成原生 HTML/CSS/JS，也可以，但建议所有 agent 在同一轮协作中保持一致，不要混用多套脚手架。

## 工程结构建议

建议尽快收敛为如下结构：

```text
src/
  app/
  components/
  features/game/
    engine/
    hooks/
    utils/
    types/
  styles/
  tests/
public/
```

关键原则：

- 游戏规则写成纯函数，避免与 UI 强耦合
- 动画表现层与棋盘数据层分离
- 随机数生成逻辑可注入，方便测试
- 本地存储统一封装，避免散落的 `localStorage` 调用

## 分阶段实施计划

### Phase 1：项目初始化

- 初始化 `Vite + React + TypeScript`
- 配置基础目录结构
- 接入 `ESLint`、`Prettier`、`Vitest`
- 建立全局设计变量与基础页面框架

完成标准：

- 本地可启动
- 有基础首页壳子
- 有可运行测试命令

### Phase 2：核心游戏引擎

- 定义棋盘数据结构
- 实现生成初始棋盘
- 实现四方向移动和合并
- 实现得分累计
- 实现无可移动时结束判断

完成标准：

- 核心规则测试覆盖关键边界
- 单步移动结果与标准 2048 规则一致

### Phase 3：界面与交互

- 渲染棋盘与数字块

- 移动端滑动手势
- 分数区、最高分、重新开始按钮

完成标准：

- 用户可完整玩一局
- 桌面与移动端都可操作

### Phase 4：体验打磨

- 增加生成、移动、合并动画

完成标准：

- 手感自然，没有明显闪烁和状态错乱
- 视觉层次和品牌感明显优于“默认模板”

### Phase 5：质量保障与发布准备

- 增补组件测试和 E2E 测试
- 检查移动端适配、横竖屏与小屏设备

- 补全文档与部署说明

完成标准：

- 核心流程可被自动化验证
- README、命令、结构、已知限制全部清晰

## 验收标准

项目完成时至少应满足：

- `2048` 规则正确，无重复合并 bug
- 新局、结束、胜利、继续游戏流程稳定
- 键盘与触摸操作都可用
- 刷新页面后最高分保留
- 界面在常见桌面和手机宽度下不破版
- 测试覆盖核心引擎与至少一条主流程

## 多 Agent 协作拆分建议

适合并行拆成以下工作流：

- Agent A：脚手架、工程化、目录结构、规范配置
- Agent B：游戏引擎与规则测试
- Agent C：UI 组件、视觉主题、动画
- Agent D：交互层，包括键盘、手势、弹层、状态切换
- Agent E：测试、回归检查、边界 bug 修复、文档补全

协作约束：

- 不要同时让多个 agent 改同一批核心文件
- 游戏引擎优先纯函数化，减少合并冲突
- UI 与引擎通过明确类型和接口协作
- 每个 agent 完成后都应自测并更新说明

## 风险清单

- 合并规则实现错误，尤其是“单轮只合并一次”
- 动画状态与真实数据不同步
- 手势方向判断不稳定
- 最高分持久化逻辑遗漏
- 快速输入导致重复触发移动
- 小屏幕布局溢出

## 建议开发顺序

1. 先搭脚手架和代码规范
2. 再做纯函数游戏引擎
3. 再接 UI 和输入控制
4. 再做动画和视觉强化
5. 最后补测试、修复和文档
