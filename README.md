## 项目目标

打造一个体验完整、视觉精致、逻辑可靠、便于扩展的新版 `2048` 游戏，至少满足以下要求：

- 支持桌面端与移动端
- 核心移动与合并规则完全正确
- 有分数、最高分、重新开始、失败检测、胜利提示
- 有平滑动画、清晰反馈和统一视觉风格
- 有可维护的代码结构和自动化测试

## 推荐技术路线

为减少分歧，建议统一采用以下技术栈：

- 前端：`Vite + React + TypeScript`
- 样式：原生 `CSS` 或 `CSS Modules`，使用设计变量管理颜色、圆角、阴影和动画时长
- 状态管理：`useReducer` 或独立的纯函数游戏引擎
- 单元测试：`Vitest`
- 组件测试：`React Testing Library`
- 端到端测试：`Playwright`
- 代码质量：`ESLint + Prettier`

如果后续要改成原生 HTML/CSS/JS，也可以，但建议所有 agent 在同一轮协作中保持一致，不要混用多套脚手架。

## 部署说明

当前仓库使用 GitHub Actions 发布 GitHub Pages：

1. 推送到 `main` 后，`.github/workflows/deploy.yml` 会执行 `npm ci` 和 `npm run build`。
2. Actions 构建时会设置 `GITHUB_PAGES=true`，让 Vite 使用 `/2048-New-era/` 作为资源 base。
3. 构建完成后仅上传 `dist` 到 GitHub Pages，线上页面不应直接加载 `/src/main.tsx`。
4. 仓库 Settings → Pages 的 Source 应选择 `GitHub Actions`。

如果线上控制台再次出现 `/src/main.tsx` 404，优先检查 Pages Source 是否仍指向分支根目录，而不是 Actions。

## 工程结构建议

建议尽快收敛为如下结构：

```text
src/
  app/
  components/
  features/game/
    engine/
    hooks/
    utils/
    types/
  styles/
  tests/
public/
```

关键原则：

- 游戏规则写成纯函数，避免与 UI 强耦合
- 动画表现层与棋盘数据层分离
- 随机数生成逻辑可注入，方便测试
- 本地存储统一封装，避免散落的 `localStorage` 调用

## 验收标准

项目完成时至少应满足：

- `2048` 规则正确，无重复合并 bug
- 新局、结束、胜利、继续游戏流程稳定
- 键盘与触摸操作都可用
- 刷新页面后最高分保留
- 界面在常见桌面和手机宽度下不破版
- 测试覆盖核心引擎与至少一条主流程

## 多 Agent 协作拆分建议

适合并行拆成以下工作流：

- Agent A：脚手架、工程化、目录结构、规范配置
- Agent B：游戏引擎与规则测试
- Agent C：UI 组件、视觉主题、动画
- Agent D：交互层，包括键盘、手势、弹层、状态切换
- Agent E：测试、回归检查、边界 bug 修复、文档补全

协作约束：

- 不要同时让多个 agent 改同一批核心文件
- 游戏引擎优先纯函数化，减少合并冲突
- UI 与引擎通过明确类型和接口协作
- 每个 agent 完成后都应自测并更新说明

## 风险清单

- 合并规则实现错误，尤其是“单轮只合并一次”
- 动画状态与真实数据不同步
- 手势方向判断不稳定
- 最高分持久化逻辑遗漏
- 快速输入导致重复触发移动
- 小屏幕布局溢出

## 建议开发顺序

1. 先搭脚手架和代码规范
2. 再做纯函数游戏引擎
3. 再接 UI 和输入控制
4. 再做动画和视觉强化
5. 最后补测试、修复和文档
